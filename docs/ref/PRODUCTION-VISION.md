## 목표

사용자가 PNG 파일 하나를 업로드하면, **가능한 한 빠르게**(현실적으로는 1~3분 내) 움직이는 버튜버 모델이 나와서 웹캠으로 방송을 바로 시작할 수 있다. 이를 위해 현재 파이프라인과의 격차를 정량화하고, 병목·기술 장애물·우선순위를 기록한다.

## "즉각"을 막는 4대 장애물

| 순위 | 장애물 | 영향 | 해결 가능성 |
|---|---|---|---|
| **1위** | **휴먼 게이트 2개 (H1_5·H2)** | 개별 시간 미측정 — 76분 중 사람 검수·개선이 ~54분 + 파이프라인 완전 차단 | 중기(자동 게이트 대체 구조 필요) |
| **2위** | 레이어 분해 ~20분(See-through ML) | 순수 연산의 대부분 | 단기(클라우드 CUDA GPU로 1~2분으로 단축 가능) |
| **3위** | 생성 단계(입·눈 API) | 재현성 없음(매번 달라짐) + 비용 + 시간 | 중기(결정론적 워프/캐시로 대체) |
| **4위** | 앵커 자동 검출 미구현 | 임의 업로드 PNG는 입력 자체가 불가 | 단기(기술적으로는 easy, 순서상 선행조건) |

## 핵심 결론

### 계산 로직은 이미 충분히 빠르다

리허설 모드(분해 재사용 + 게이트 스킵)에서 **1분 38초**만에 완주하는 것으로 증명된 바, 앵커·리깅·검증 로직 자체는 병목이 아니다. 지연의 전부는:

- **ML(분해)**: 맥 MPS 640 해상도, 20분 — 클라우드 CUDA로 1~2분으로 단축 가능
- **API(생성)**: 이미지 생성 시간 + 비용 + 재현성 없음
- **사람(게이트)**: 배치 확인·최종 승인 10~15분 + 파이프라인 완전 차단

### "수초 즉각"은 본질적으로 불가능

ML 분해가 20분이고, 그 결과가 없으면 리깅을 시작할 수 없다. 클라우드 가속해도 1~2분. 생성 API도 수분. 따라서 **"수초"는 기술 한계이다.**

### "업로드 → 1~3분 → 모델 + 진행 표시"는 충분히 도달 가능

분해를 클라우드로 가속하고, 게이트를 자동화하거나 병렬/비동기로 처리하면:
- 분해: 1~2분(CUDA, 추정)
- 생성: 수분(미측정, 병렬화 가능)
- 리깅: 1분 38초 내(기존 verified)
- 게이트: 자동 + 진행바(UX로 "즉각 느낌" 제공)

**합계: 1~3분 = 현실적 프로덕션 목표**

### 게이트 자동화 로드맵 (2순위, 2026-06-15부터 시작)

2026-06-15에 추가한 **"입 함정 방어 건강성 보강 3갭"** 이 이 단계의 첫 단추다:

- **(A) SHEET-P0 피부 면적 게이트**: 생성 입 시트에 얼굴 윤곽이 그려지면(전경 살구색>80%) FAIL → 재생성
- **(B) 4상태 개구상태 품질 게이트**: 입의 윗입술·아랫입술·내부 비율이 기준치를 벗어나면 FAIL
- **(C) 부품 하한 + 게이트 비율 기록화**: `upper_lip` 최소 픽셀 수·각 부품 비율 표준화

이 게이트들이 촘촘해질수록 H2(사람 시각 판정)를 자동으로 대체 가능해진다. 상세 내용은 evidence `AUTORIG-CHARACTER-005`에 기록.

## 연결점: 2026-06-15 입 함정 방어 ↔ 로드맵 2순위

**문제**: 004 위벨(입 5번 삽질) → 005 지피쨩(입 부품 분해 오인)으로 유사 함정이 반복됐다.

**해결책**: H2(주인님)의 "입이 이상하면 FAIL"이라는 판정을 자동 게이트가 선제적으로 catch하려면, 입의 가능한 모든 결함을 **수치**로 정의해야 한다.

- 004 반성: "입 시트에 얼굴이 그려져서"(생성 특성) + "앵커 오정렬"(추출 설정) + "아랫입술 비율 부족"(부품 체계)이 결합해 진단 어려움.
- 005 입 함정 방어(2026-06-15): SHEET-P0·4상태 게이트·부품 비율 기록 추가 → 다음 캐릭터들이 이 함정을 건너뛸 수 있음.
- **그 다음(로드맵 2순위)**: 이 자동 게이트들이 충분히 촘촘해지면, H2 게이트 자체를 대체하는 "신뢰도 점수" 시스템으로 전환 가능.

최종 목표: **"게이트 단계가 끝나면 거의 항상 H2 승인 나옴"** = 사람 게이트가 형식적 재확인 수준으로 축소.


## 정적 배포 검증 (2026-06-15, PoC 입증)

### 가설
005 모델 뷰어(PixiJS+MediaPipe 드라이브 앱)가 동적 python 서버 없이 순수 정적 호스팅만으로 작동하는가.

### 검증 방법
`mini_cubism_app/`(index.html + vendor/pixi.min.mjs + src/) 정적 복사 + rig의 parts PNG 46장을 `project/parts/`로 복사 + `/api/project` 응답(character.json + mini_rig.json + `_project_base_url`)을 정적 JSON으로 구워서, **순수 `python -m http.server`**(프로젝트 고유 서버 로직 0)로 띄움.

### 검증 결과
**성공.** 브라우저에서 005 지피쨩이 PixiJS로 정상 렌더됨(에러 0, canvas 생성). `/`·`/api/project`·`/project/parts/*.png` 모두 HTTP 200.

### 결론
모델 뷰어는 동적 백엔드가 전혀 필요 없다 → Vercel·Netlify·itch.io·GitHub Pages 등 정적 호스팅에 그대로 배포 가능. 동적인 것은 에디터 저장·검증 POST(`/api/mini-rig`, `/api/validate-*`)뿐이고 방송용엔 불필요.

### 정적 빌드 레시피 (확정)
① `mini_cubism_app/` 복사  
② parts PNG → `project/parts/`  
③ `/api/project` = character.json + mini_rig + `_project_base_url:"/project/"` 정적 JSON으로 굽기.

## 웹검색 결론 — 쉬운 배포 업계 표준 (2026-06-15)

### 배포 표준
"바로 쓰게 쉽게 주는" 업계 표준 = 정적 웹앱 + 무료 호스팅(Vercel 등) + 공유 링크 하나 + OBS 연동.

OBS 연동은 투명 배경보다 **green screen 토글 + OBS 크로마키**가 검증된 실용법(브라우저 투명 캡처는 환경 의존적).
